### 1.2 关键修正说明

| 项目 | 原方案（错误） | 修正方案 |
|------|--------------|---------|
| 有线输出 | 通过CH585 USB FS | **H417 USB3.0直接→PC** |
| 有线回报率 | 1000Hz | **32KHz（USB3.0带宽支撑）** |
| CH585角色 | 三模全部 | **仅负责BLE+2.4G无线** |
| H417-CH585接口 | 未明确 | **USB2.0 HS 480Mbps** |
| 无线回报率 | 未明确 | **1/2/4/8K可调** |

### 1.3 三模输出架构

```
键盘按键 → MUX/ADC → H417处理
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
         USB3.0        USB HS      本地消费
       (有线32K)    → CH585       (H417 OS)
                          │
                    ┌─────┴─────┐
                    ▼           ▼
                 BLE 5.4     2.4G私有
                 (1/2/4/8K)  (1/2/4/8K)
```

**模式优先级**：有线(USB3.0) > 2.4G > BLE。插入USB线时自动切换有线模式；无线模式下优先使用2.4G（延迟更低），BLE作为备用。

**回报率配置**：

| 连接方式 | 可选回报率 | 默认 | 说明 |
|----------|-----------|------|------|
| USB3.0有线 | 1K/2K/4K/8K/**32K** | 8K | USB3.0带宽充足，32K极限模式 |
| 2.4G无线 | 1K/2K/4K/**8K** | 4K | CH585 2.4G协议上限8K |
| BLE无线 | 125Hz/250Hz/**500Hz**/1K | 500Hz | BLE 5.4理论上限，兼顾功耗 |

### 1.4 CH585 通信协议

H417通过USB2.0 HS Host与CH585通信，协议设计：

**上行管道（CH585→H417）**：无线状态报告
- 端点：Interrupt IN
- 包格式：`[状态码(1B)][设备ID(1B)][RSSI(1B)][电池电量(1B)][配对列表(8B)]`
- 频率：1Hz（常态），10Hz（配对/切换时）

**下行管道（H417→CH585）**：控制命令
- 端点：Control / Interrupt OUT
- 命令集：
  - `0x01` 启动BLE广播
  - `0x02` 连接指定设备
  - `0x03` 断开当前连接
  - `0x04` 切换到2.4G模式
  - `0x05` 设置2.4G频道
  - `0x06` 开启/关闭配对模式
  - `0x10` 查询配对设备列表
  - `0x20` 设置BLE连接参数（间隔/延迟/超时）
  - `0x30` 进入低功耗模式

### 1.5 键盘事件路由

H417固件中维护一个路由决策模块：

```
按键事件 ──► [路由决策器]
                │
    ┌───────────┼───────────┐
    │           │           │
    ▼           ▼           ▼
 USB3.0      无线协议    本地Input
 HID报告    (BLE/2.4G)   子系统
 → PC         → PC/手机   → H417 OS
```

**路由规则**：
- USB3.0线插入 → 全部事件走HID报告到PC
- 无线模式+本地屏不活跃 → 全部事件走无线到PC
- 无线模式+本地屏处于输入焦点 → 事件同时到本地OS和PC（或根据设置仅到本地）
- KVM切换键 → 切换目标设备，事件路由到指定通道

## 二、自研键盘固件架构

### 2.1 为什么不用QMK/VIA

| 维度 | QMK/VIA | 自研方案 |
|------|---------|----------|
| 磁轴模拟输入 | 需大幅改造ADC层 | 原生适配H417的16路ADC+DMA |
| 32K回报率 | USB协议栈需优化 | USB3.0 OTG直接控制 |
| 三模切换 | 不支持无线协议栈 | CH585独立处理BLE+2.4G |
| 本地事件消费 | 无此概念 | H417 OS直接接收input事件 |
| 屏幕联动 | 无 | 按键状态实时显示在480x800屏 |
| 旋钮/摇杆 | VIA支持有限 | 自定义传感器框架 |
| 代码体积 | 完整固件150KB+ | 核心50KB，精简高效 |
| 启动速度 | 数秒 | 毫秒级冷启动 |

### 2.2 固件分层

```
┌────────────────────────────────────────┐
│ 应用层：AI终端界面 / 本地命令行          │  ← V5F
├────────────────────────────────────────┤
│ 路由层：HID输出 / 无线发送 / 本地分发     │  ← V5F
├────────────────────────────────────────┤
│ 逻辑层：键值映射 / 层切换 / 宏 / DKS    │  ← V5F
├────────────────────────────────────────┤
│ 磁轴引擎：RT / 触发花样 / 模拟输入       │  ← V5F
├────────────────────────────────────────┤
│ IPC Mailbox ←── 共享SRAM ──►           │
├────────────────────────────────────────┤
│ 采集层：MUX扫描 / ADC / DMA / 校准      │  ← V3F
├────────────────────────────────────────┤
│ 设备层：GPIO / USB3.0 OTG / USB HS Host │  ← V3F/V5F
├────────────────────────────────────────┤
│ 扩展层：EC11 / 摇杆ADC / WS2812 / I2C   │  ← V3F
└────────────────────────────────────────┘
```

### 2.3 磁轴引擎设计

**触发算法清单**：

| 算法 | 说明 | 适用场景 |
|------|------|----------|
| 静态触发点 | 固定深度触发（0.1~4.0mm可调） | 日常打字 |
| Rapid Trigger (RT) | 抬起即重置，0.04mm精度 | FPS游戏 |
| 动态键程 DKS | 单键4个触发点，4种不同输出 | 复杂宏 |
| 双效点击 Mod Tap | 单击=A功能，长按=B功能 | 空间受限 |
| Speed Tap / FlashTap | A+D同时按下，优先最后输入 | FPS移动 |
| 模拟输入 | 按键深度→模拟值（如手柄摇杆） | 赛车/飞行 |
| SOCD清理 | 互斥键同时按的优先级规则 | 格斗游戏 |
| 连发 Turbo | 按住自动连击，频率可调 | 连射操作 |

**触发花样**：每个按键可独立配置触发模式，支持8组板载配置文件切换。

**性能开销**：磁轴引擎运行在整个扫描循环中，V3F以~1MHz频率执行，处理全部磁轴算法占用<15% CPU。

### 2.4 扩展设备支持

| 设备 | 接口 | 驱动方式 | 功能 |
|------|------|----------|------|
| EC11旋钮 | GPIO中断+定时器 | 旋转编码器状态机 | 音量/滚动/切换 |
| 模拟摇杆 | ADC通道 | 死区处理+方向量化 | 方向控制/菜单导航 |
| WS2812灯带 | SPI/GPIO+DMA | 时序精确控制 | RGB灯效（每灯独立寻址） |
| 触摸条 | ADC或TouchKey | 滑动检测+位置计算 | 音量条/进度条 |

### 3.2 磁轴专用功能深度对比

#### ROG Armoury Crate（奥创 + 极速版）

ROG为磁轴键盘推出了独立的"奥创极速版"ARMOURY CRATE GEAR，精简了主软件的功能臃肿问题[^757^][^760^][^761^]。

核心功能：
- 触发行程0.1-4mm，40段调节
- Rapid Trigger独立开关+每键可调
- Speed Tap：A/D同时按下时优先最后输入，自动释放前一个
- 单键MT（Mod Tap）、DKS（动态键程）
- 连发设置
- 触控SmartBar多功能切换（音量/媒体/亮度/触发点/自定义）
- Aura Creator高级灯效编辑器
- AURA Sync跨设备灯效同步
- 网页驱动（规划中）

#### Wootility（行业标杆）

Wootility是磁轴键盘配置软件的行业标杆，以简洁直观著称[^749^][^750^][^668^]。

独特功能：
- **Rappy Snappy**：两个按键永不同时触发，仅保留按压更深的那个
- **Snappy Tappy (SOCD)**：自定义两键同时激活时的优先级规则（最后优先/X优先/Y优先/中立模式）
- **动态键程 DKS**：单键4个触发点，可分配4种不同操作（按下/深按/释放/完全松开）
- **手柄模式**：WASD键获得手柄摇杆般的线性模拟输入
- **组合键Combo Key**：将常用操作序列绑定到单键（如移动+冲刺）
- **双效点击Mod Tap**：单按=A功能，长按=B功能
- 客户端+浏览器双模式
- 板载4套配置

#### Corsair iCUE

iCUE以灯光生态著称，磁轴功能集中在"键启动"页面[^813^][^816^][^820^]。

核心功能：
- 触发行程0.4-3.6mm（相对保守）
- **双点触发**：一次按键可输出两个指令（轻按=走，重压=跑）
- Rapid Trigger集成键程控制和复位精细调整
- 灯光层叠加（多区域独立灯效）
- 桌面壁纸颜色同步
- 电竞模式按钮（硬件级抗干扰+红色灯效）
- AXON技术8000Hz
- 板载**50套配置文件**
- FlashTap技术（SOCD变体）

#### SteelSeries GG / Engine

赛睿Apex Pro的OLED屏幕是其最大特色[^808^][^811^][^826^]。

独特功能：
- **OLED屏幕自定义**：可显示自定义图形、GIF、系统信息（CPU/GPU使用率）
- 游戏内HUD联动：OLED显示游戏内信息
- 双触发（两段式触发，轻按/重压不同功能）
- OmniPoint 2.0轴0.2-3.8mm可调
- SteelSeries Engine功能全面，每键独立调节RT
- AIDA64系统集成：OLED实时显示PC硬件状态

#### 迈从M HUB（网页驱动代表）

迈从M HUB代表了国产键盘驱动的新趋势[^800^][^801^][^802^]。

特点：
- **客户端+网页版双模式**，网页版无需安装
- 独立RT按键，快速切换3组板载配置
- 多功能旋钮支持
- 轻量设计

## 四、AI功能嵌入设计

### 4.1 竞品现状

目前没有任何键盘配置软件集成AI功能。这是一个**空白领域**。AI的嵌入可以从以下维度切入：

### 4.2 AI嵌入方案

#### 方案A：智能配置推荐

**原理**：通过桥接服务器分析用户的AI Agent使用模式，推荐最优键盘配置。

示例：
- 检测到用户频繁使用Codex CLI → 自动推荐"AI开发"配置文件（宏绑定常见命令）
- 检测到用户在玩Valorant → 推荐"FPS游戏"配置（WASD触发点0.1mm+RT）
- 检测到用户长时间打字 → 推荐"打字"配置（触发点2.0mm，防误触）

**实现**：配置软件调用桥接服务器的用户行为API，获取应用使用统计，匹配预设规则库。

#### 方案B：自适应触发校准

**原理**：AI分析用户的打字习惯，动态微调每个按键的触发点。

- 某按键频繁误触 → AI建议提高该键触发点0.2mm
- 某按键感觉延迟 → AI建议降低触发点
- 用户在游戏中频繁双击失败 → 自动优化RT重置距离

**实现**：固件收集按键统计数据（误触率、平均按压力度、双击成功率），上传云端分析，回推建议到配置软件。

#### 方案C：AI Agent快捷面板

**原理**：在配置软件中集成AI Agent控制面板，与键盘快捷键绑定。

功能：
- 一键绑定"确认当前AI操作"到指定按键
- 一键切换AI Agent模式（安全/常规/自主）
- 可视化显示Agent任务列表，点击跳转详情
- 宏录制支持插入"等待AI确认"步骤

**实现**：配置软件通过WebSocket连接到桥接服务器，实时同步Agent状态。

#### 方案D：自然语言配置

**原理**：用户用自然语言描述需求，AI自动生成对应配置。

示例：
- "我要打瓦的急停配置" → 自动设置WASD触发点0.1mm+RT 0.1mm+SpeedTap
- "帮我设置一个AI开发模式" → 自动创建层：F1=确认AI操作，F2=暂停Agent，F3=查看任务
- "灯效改成呼吸蓝" → 直接修改灯效参数

**实现**：集成LLM API（本地或云端），解析用户意图，映射到配置参数。

## 7. 传感器扩展与交互设计

AI终端的人机交互质量取决于传感器层的完整性与协同效率。CH32H417双核RISC-V微控制器（MCU）提供4组I²C、4组SPI、2组I²S、DVP数字图像接口及16路TouchKey电容触摸检测通道[^10^][^110^]，为构建多模态交互系统提供了充足的物理连接资源。本章基于硬件接口能力，给出传感器选型建议、BOM成本分析及交互架构设计。

### 7.1 推荐传感器清单

#### 7.1.1 触摸：GT911电容触摸控制器

用户已配备2.4寸480×800竖屏，需配套电容触摸控制器实现触控交互。推荐汇顶科技（Goodix）GT911，该芯片支持5点同时触摸，配置26路驱动通道与14路感应通道，扫描频率100Hz，I²C通信速率最高400kHz，Normal Mode响应时间低于48ms[^119^][^155^][^157^]。GT911支持自动跳频与温漂补偿，上电自动校准，可通过INT引脚输出中断信号，有效降低主控MCU的轮询开销。封装为QFN52（6mm×6mm），支持7~8寸面板，完全覆盖2.4寸需求[^119^]。作为替代方案，敦泰科技FT6336U采用自电容技术，支持单点/手势或两点触摸，封装更小巧（QFN48，5×5mm）[^147^][^159^]，适合对体积有严苛要求的场景。

#### 7.1.2 物理交互：EC11旋转编码器+按键矩阵

在触屏之外，物理旋钮提供精确的盲操能力，这对桌面场景中的快速参数调节至关重要。EC11旋转编码器输出AB相正交编码信号并集成按键开关，每转产生20个脉冲（单边沿计数）或40个脉冲（双边沿计数），仅需2路GPIO捕获AB相边沿变化，另1路GPIO检测按键状态[^47^][^55^]。软件层面建议采用状态机处理格雷码序列，结合硬件RC滤波与软件消抖，确保旋转方向判断的可靠性[^47^]。EC11在菜单导航、音量控制等场景中表现优异[^49^][^51^]。

按键矩阵方面，CH32H417内置16路TouchKey电容触摸检测功能[^110^]，可直接实现无物理接触的虚拟按键，减少外壳开孔。建议配置1个EC11旋钮（旋转+按压双重功能）+ 2个TouchKey虚拟按键（唤醒键、返回键），形成最小化的物理交互组合。

#### 7.1.3 环境感知：BME280与VEML7700

环境感知能力使AI终端能够根据使用场景自适应调整行为。BOSCH BME280是一款温度-湿度-气压三合一数字传感器，温度量程-40°C至+85°C（精度±1°C），湿度0%~100%RH（精度±3%RH），气压300~1100hPa（精度±1hPa），工作电流仅2.8μA（1Hz采样，正常模式），睡眠模式低至0.1μA[^196^][^200^][^194^][^205^]。BME280内置温度补偿算法，通过气压数据可计算海拔高度（精度约±1m）[^113^][^115^]，适合环境感知与定位增强场景。

光照检测推荐Vishay VEML7700，其采用Filtron技术使光谱响应接近人眼感知，量程0~120Klx（为BH1750的2倍），分辨率0.0036lx/ct，正常工作电流仅45μA[^81^]。VEML7700的输出可用于自动调节屏幕背光亮度，在暗光环境降低显示功耗，强光环境提升可读性。相较之下，BH1750量程仅1~65535lux且工作电流0.12mA[^76^][^79^]，VEML7700在量程与功耗两个维度均具优势。

#### 7.1.4 运动感知：MPU6050六轴IMU

MPU6050集成3轴加速度计（量程±2g/±4g/±8g/±16g）与3轴陀螺仪（量程±250°/s~±2000°/s），16位ADC分辨率，内置数字运动处理器（DMP）可直接输出四元数，减轻主控MCU的姿态解算负担[^59^][^61^]。待机电流仅5μA，批量单价约￥6[^50^][^69^]。MPU6050支持多种交互手势：摇晃唤醒/撤销操作、倾斜翻页、双击确认、静止自动休眠等[^221^]，显著丰富了终端的物理交互维度。若对噪声性能有更高要求，BOSCH BMI270是升级选项，其加速度计噪声160μg/√Hz、陀螺仪噪声0.007dps/√Hz，待机电流3.5μA，但单价约￥79[^53^]，成本增加约13倍。

#### 7.1.5 音频输入：INMP441 I²S数字麦克风

语音交互是AI终端的核心输入通道。INMP441是一款MEMS全向数字麦克风，内置ASIC处理电路直接输出I²S数字信号，省去外置ADC与运放[^210^]。其关键参数包括：信噪比61dB（A-weighted）、灵敏度-26dBFS@94dB SPL、采样率最高48kHz、频率响应60Hz~15kHz、声学过载点122dB SPL、典型工作功耗250μA[^210^]。封装尺寸3.5mm×2.65mm×0.98mm，模块单价约￥12[^199^][^205^]。CH32H417内置2组I²S接口，可直接连接INMP441实现音频采集[^10^]，配合400MHz双核RISC-V的处理能力，可在本地运行轻量级关键词唤醒（KWS）算法，再由云端处理自然语言理解[^195^][^214^]。

以下表格汇总推荐传感器的BOM成本与接口分配：

| 传感器 | 型号 | 接口 | 数量 | 预估单价 | 小计 | 核心功能 |
|:---|:---|:---|---:|---:|---:|:---|
| 电容触摸控制器 | GT911 [^119^] | I²C1 | 1 | ￥8 | ￥8 | 5点触控，手势识别 |

### 7.2 交互设计

#### 7.2.1 多模态唤醒：接近传感器+语音+按键

单一唤醒方式难以覆盖多样化的使用场景。推荐采用分层唤醒策略：VCNL4040接近传感器作为第一层检测，在用户手部靠近设备（<10cm）时触发中断，MCU从深度休眠进入浅休眠状态，屏幕点亮至待机界面[^201^][^202^]；EC11旋钮长按2秒或INMP441检测到唤醒词作为第二层触发，进入活跃交互状态[^195^]；MPU6050检测摇晃动作作为备用唤醒通道[^221^]。

功耗管理方面，深度休眠状态下仅VCNL4040保持工作（功耗<0.5mW），浅休眠状态增加MPU6050运动检测（总功耗约2mW），活跃状态下全部模块开启（功耗200~500mW）。30秒无操作自动回退至浅休眠，进一步降低待机功耗。SK6812 RGB LED的状态指示也需纳入功耗预算：实测传统呼吸模式下LED环平均电流18.7mA（93.5mW@5V），而POWERMANAGEMENT模式下仅0.012mA（0.06mW）[^74^]，建议通过MOSFET控制LED供电，休眠时物理断电。

#### 7.2.2 2.4寸竖屏UI架构：三区设计

480×800分辨率的2.4寸竖屏（像素密度约194 PPI）适合清晰文字与图标显示，但有效显示面积有限，信息层级的设计尤为关键。参考LVGL嵌入式GUI框架的设计原则[^231^]，建议采用状态栏+内容区+操作区的三区架构：顶部状态栏高度30像素，显示电量、WiFi/以太网连接状态与时间；中部主内容区为核心显示区域，承载对话气泡、环境数据可视化或菜单列表；底部操作区高度60像素，布置3~4个图标按钮。最小触摸区域不小于44×44像素（约5.8mm），正文字体不小于16像素，标题不小于24像素[^231^]。过渡动画时长控制在200~300毫秒，避免复杂效果占用400MHz CPU资源。结合EC11旋钮的交互映射：顺时针旋转向下滚动或增大参数，逆时针旋转向上滚动或减小参数，短按确认，长按2秒返回上级菜单[^49^]。

#### 7.2.3 参考Rabbit R1与Stream Deck的设计教训

2024年发布的Rabbit R1是当年最受欢迎的AI硬件，售价$199，其交互设计提供了重要参考。Rabbit R1采用2.88英寸触摸屏，侧边配置一个圆形按键作为一键唤醒AI的快捷键，配合滚轮浏览内容替代触屏滑动，并支持横向旋转屏幕调出键盘（利用滚轮选择字符）[^117^][^126^]。其设计验证了"物理旋钮+触屏+语音"多模态交互在AI终端中的用户接受度。Humane AI Pin则走了相反路线——无屏幕、纯语音+手势交互，辅以激光投影到手掌作为显示[^143^][^145^]，但由于缺少物理反馈且需手动点击触摸板激活[^146^]，用户体验口碑不佳。

Stream Deck在直播控制领域证明了一个关键设计原则：专用硬件在特定场景下的效率远高于通用设备（手机/电脑）。Stream Deck通过物理按键提供即时、无需视觉确认的操作，这一"Companion设备"定位与AI终端的桌面场景高度吻合[^117^]。

以下表格对比三种交互范式的关键维度：

| 对比维度 | Rabbit R1 [^117^][^126^] | Stream Deck | 本方案（CH32H417 AI终端） |
|:---|:---|:---|:---|
| 屏幕尺寸 | 2.88寸触摸屏 | 无（LCD按键屏） | 2.4寸480×800竖屏 |
| 物理输入 | 滚轮+侧边AI键 | 15个可编程按键 | EC11旋钮+TouchKey |
| 语音输入 | 支持 | 不支持 | INMP441 I²S麦克风 |
| 唤醒方式 | 按键+语音 | 即时按键 | 接近检测+按键+语音 |
| 环境感知 | 无 | 无 | BME280+VEML7700+VCNL4040 |
| 手势控制 | 无 | 无 | MPU6050摇晃/倾斜 |
| 状态指示 | 屏幕+LED | LCD按键自发光 | SK6812 RGB灯环 |
| 典型功耗 | ~500mW（联网） | ~200mW | 休眠<0.5mW / 活跃200~500mW |
| 定价 | $199 | $149起 | 预估$49~89 |

上表揭示了几条值得吸取的设计教训。第一，Rabbit R1的滚轮导航在内容浏览场景中效率高于纯触屏滑动[^126^]，这直接支持本方案引入EC11旋钮的决策。第二，Stream Deck的"即时操作、无需视觉确认"理念[^117^]说明物理按键/旋钮在Companion场景中的不可替代性——用户不需要盯着设备即可完成常用操作。第三，Humane AI Pin的困境表明，完全依赖语音+手势而缺乏视觉反馈的交互模式在当前技术条件下尚不成熟[^143^]。本方案在Rabbit R1的多模态基础上，增加了环境感知（温湿度、气压、光照）与运动感知（IMU手势）两个维度，使终端能够根据环境条件自适应调整行为（如自动调节屏幕亮度、检测用户靠近自动唤醒），这是现有AI硬件产品均未覆盖的差异化能力。预计这种"环境感知+多模态交互"的组合，将显著提升终端在日常桌面场景中的使用粘性与用户体验连贯性。

## 9. 实施路径与风险

基于前文对硬件能力、软件架构和竞争格局的系统分析，本章提出一个分三阶段推进的实施路径，并对每一阶段的关键交付物、时间估算与技术风险进行量化评估。整体项目周期预计为6-7个月，每阶段均设置明确的可验证里程碑，以降低不确定性并支持早期市场验证。

### 9.1 分阶段实施

#### 9.1.1 第一阶段（第1-3月）：终端模拟器 + WebSocket + MVP验证

第一阶段的核心目标是在最短时间内构建一个可工作的最小可行产品（Minimum Viable Product, MVP），验证"AI Agent硬件监控"这一产品假设。技术路线上，该阶段聚焦三项并行工作：终端模拟器引擎、WebSocket通信栈和桥接服务器原型。

终端模拟器是MVP的UI核心。调研显示，VT100/ANSI终端模拟器在8位MCU（如ATMega）上已有成熟实现 [^131^]，在STM32F103（72MHz Cortex-M3）上的移植版本仅需45.9KB代码空间 [^186^]。CH32H417的400MHz V5F内核配合896KB SRAM，预计可实现60 FPS以上的文本刷新率，CPU占用率低于25% [^3^]。终端模拟器采用脏矩形（Dirty Rectangle）优化策略，仅重绘变化的字符区域，可显著降低SDRAM带宽消耗。这一设计选择使得第一阶段无需等待完整的LVGL移植，大幅缩短了开发周期。

通信栈方面，设备端需要实现一个轻量级WebSocket客户端（基于libwebsockets或类似的C库），通过JSON Lines格式与桥接服务器交换数据 [^63^][^53^]。桥接服务器运行在PC端，负责将Codex CLI的`codex remote-control`输出 [^206^] 或Claude Code的`--output-format stream-json`流 [^168^] 转换为统一的JSON Lines事件流，供设备消费。Happy项目采用的"CLI包装器+中继服务器+客户端"三层架构 [^86^] 为桥接服务器的设计提供了直接参考。

第一阶段里程碑为：设备能够通过终端界面实时显示AI Agent状态（运行中/等待/完成），并支持一键批准或拒绝权限请求。此阶段面向早期开发者用户收集反馈，验证产品价值主张。

#### 9.1.2 第二阶段（第4-5月）：LVGL GUI + 图形化状态显示

在MVP验证通过后，第二阶段的目标是将纯文本终端升级为图形化界面。该阶段的核心工作是RT-Thread标准版移植、LVGL v9集成以及GPHA图形加速器的适配。

RT-Thread官方BSP支持CH32H417 [^557^]，且MounRiver Studio II内置了RT-Thread工程模板 [^430^]，预计操作系统移植与基础驱动适配可在2-3周内完成。LVGL v9在400MHz RISC-V平台上的性能有明确对标：ESP32-P4（同频双核RISC-V）在480×480分辨率下可达约90 FPS，在1024×600分辨率下为58-62 FPS [^51^]；STM32H723（550MHz Cortex-M7）在800×480分辨率下可达82 FPS [^269^]。CH32H417在480×800分辨率（38.4万像素，介于上述对标之间）下，预估可实现45-60 FPS的中等复杂度UI渲染 [^3^]。

GPHA（Graphics Processing Hardware Accelerator）的适配是该阶段的最大技术变量。若GPHA可用，参考STM32 DMA2D可将渲染帧率从15-20 FPS提升至40+ FPS [^98^]，ESP32-P4的2D-PPA在纯填充场景下速度提升可达9倍 [^101^]。然而GPHA的公开文档极为有限 [^5^][^54^]，因此开发团队必须在启动LVGL移植的同时，并行准备纯软件渲染的fallback方案——基于V5F内核的软件渲染能力，即使无硬件加速，预估仍可维持30+ FPS的基本可用水平 [^61^]。

第二阶段的交付物为一个完整的图形化AI Agent监控界面，包含任务状态卡片、实时日志滚动视图和快捷操作面板，所有内容通过WebSocket实时同步。

#### 9.1.3 第三阶段（第6-7月）：MicroPython + 传感器 + 用户脚本

第三阶段面向生态建设，核心目标是通过脚本语言支持降低用户二次开发门槛，形成社区驱动的功能扩展体系。

MicroPython的移植在该平台上具有高度可行性。CH32V307（144MHz RISC-V，64KB SRAM）已成功运行MicroPython [^117^][^120^]，CH32H417的896KB SRAM是前者的14倍，运行环境更为充裕。RT-Thread内置MicroPython软件包 [^283^]，进一步降低了移植工作量。Lua是另一个备选方案，核心实现仅需约25KB RAM [^285^]，适合作为轻量级脚本层。此外，64MB SDRAM（512Mbit）使得设备可以容纳完整的Python标准库子集，用户可编写自定义脚本实现传感器数据读取（通过CH32H417的I2C/SPI接口连接温湿度、光线等传感器）、界面定制和自动化工作流。

该阶段同时实现V3F内核的独立应用场景。V3F（150MHz）作为I/O处理器，可在MicroPython运行时之外独立管理键盘扫描、USB事件和传感器中断，确保实时性任务不受脚本执行阻塞。

### 9.2 关键风险

| 风险项 | 影响程度 | 发生概率 | 缓解策略 | 残余风险 |
|:---|:---:|:---:|:---|:---:|
| GPHA文档不足，硬件加速无法启用 | **高** | 中 | ① 移植初期即准备纯软件渲染fallback，V5F@400MHz软件渲染预估可达30+ FPS [^61^]；② 主动联系WCH获取GPHA寄存器手册；③ 参考STM32 DMA2D编程模式逆向推断 | 低 |
| RISC-V第三方库/中间件数量不足 | **中** | 高 | ① 通过MicroPython/Lua引入高级语言生态 [^117^][^285^]；② 利用RT-Thread 450+软件包补充中间件 [^283^]；③ 64MB SDRAM（512Mbit）弥补生态短板，可直接移植POSIX兼容项目 | 低 |
| AI Agent CLI API不兼容变更 | **中** | 高 | ① 桥接服务器层统一抽象Agent协议，设备端仅消费标准化的JSON Lines事件流 [^63^]；② 支持多Agent后端（Codex/Claude/MCP）的插件化适配 [^99^]；③ 事件格式版本化设计 | 极低 |
| RT-Thread双核SMP适配延迟 | **中** | 中 | ① 采用V5F运行RT-Thread标准版+V3F裸机/Bare Metal的异构模式 [^165^]；② 自定义共享内存+IPC中断实现核间通信 | 低 |
| SDRAM信号完整性与带宽瓶颈 | **中** | 低 | ① 参考官方评估板PCB布局 [^339^]；② 使用RGB565（16bpp）降低带宽需求至46 MB/s [^203^]；③ 双缓冲放置在不同SDRAM Bank减少冲突 [^210^] | 极低 |
| AI Agent市场热度下降 | **高** | 中 | ① 产品同时支持通用SSH终端和系统监控功能，不绑定单一Agent；② 开源社区驱动功能迭代，降低对特定AI工具的依赖 [^573^] | 低 |

上表梳理了项目实施过程中六类关键风险及其量化评估。从技术风险维度看，GPHA文档不足被识别为影响程度最高的单一风险项——一旦硬件加速不可用，图形渲染性能将直接下降约40-60%。但交叉验证表明，纯软件渲染在400MHz RISC-V上仍可达到功能演示水平 [^61^]，且LVGL的脏矩形机制能显著降低全屏重绘需求 [^105^]，因此该风险的残余影响可控。RISC-V生态成熟度（综合评分3.5/5 [^5^]）是另一个持续存在的背景性风险，但64MB SDRAM（512Mbit）的"内存换生态"策略——通过运行解释器获得高级语言生态——已被ESP32的MicroPython成功实践所验证 [^117^]。

从市场风险维度看，AI Agent CLI的API变动频率是该领域固有特征：Codex CLI在过去6个月内新增了`remote-control` [^206^] 和`exec-server` [^206^] 等多个子命令，Claude Code的`--sdk-url` WebSocket协议也经历了逆向工程驱动的格式演变 [^177^][^194^]。桥接服务器架构的核心价值正是在设备端与Agent协议之间建立隔离层——设备端始终通过统一的JSON Lines事件流消费数据，桥接服务器负责处理各Agent后端的协议差异和版本适配，这一设计使得上游API变动对设备固件的影响降至最低。
